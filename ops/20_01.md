## Phase 1 — Evidence alignment + metric definition hygiene (P0)

Goal: Ensure the app’s clinical metric labels and constraint values match the evidence pack and are internally consistent.

Tasks

1. Open `docs/paper/Supplementary-Table-S1.pdf` and extract the exact constraint values used in the tool:

   * Spinal cord max constraint
   * Spinal cord PRV max constraint
   * Oesophagus Dmean
   * Lung minus GTV Dmean, V16, V4.5 (including “1st/2nd” tiers)
   * Heart V19.5 (if achievable), V39, Dmean (note any ambiguity)
2. Audit the UI and rubric:

   * If the UI currently shows **D0.03cc** but the paper specifies **Dmax**, decide the correct approach:

     * Default safe rule: align the app to the paper’s wording (Dmax).
     * If you retain D0.03cc as an internal proxy, you must: (a) label it clearly, (b) document rationale, and (c) confirm thresholds are explicitly approved (otherwise do not proceed).
3. Fix any rounding drift (e.g., 30.8 vs 30.9; 33.9 vs 34.0). Display should match the source (with consistent precision).
4. If heart Dmean tiers are ambiguous, mark that metric as **“Not scored (pending adjudication)”** and set its weight to 0 until clarified.

Implementation notes

* Put extracted constraints in a single rubric source file (e.g., `src/rubric/rubric.ts`) with `rubricVersion`.
* Add a short “Evidence source” comment per constraint pointing to `docs/paper/Supplementary-Table-S1.pdf`.

Outputs

* Update rubric/config and UI labels for metric definitions and limits.
* Add/extend unit tests verifying:

  * constraints are loaded as expected
  * critical-fail thresholds behave correctly

Acceptance criteria

* Metric names and displayed limits match Table S1 exactly (unless explicitly documented and approved).
* Tests prove the critical-fail logic works.
* Any ambiguous metric is explicitly unscored with a visible UI note.

Stop/ask if

* You cannot reconcile Dmax vs D0.03cc without a project decision. Provide a recommendation and halt further rubric changes.

Outputs

* `src/theme/tokens.*` (or equivalent)
* `src/styles/global.css` (or equivalent) updated to use tokens
* Minimal component primitives or utility classes if used (without adding deps)

Acceptance criteria

* Visual system clearly “reads” as a modern healthcare product, not a prototype.
* Focus rings are visible and consistent.
* No external font/CDN dependency introduced.
* Build passes.

* SharePoint policy constraints require a specific CSS approach.

## Phase 4 — Rebuild Calculator as a high-density scoring table (core value)

Goal: Make Calculator a fast instrument: one screen, minimal scrolling, instant feedback, sticky summary.

Tasks

1. Replace the current stacked input sections with a **table**:
   Columns:

   * Metric (name + brief definition tooltip)
   * Value (input)
   * Units
   * Constraint / Goal (display tiers if relevant)
   * Points (numeric)
   * Status (Pass/Warn/Fail chip)
2. Add a **sticky right rail** (desktop):

   * Total score
   * Subscores by category (Target/Cord/Oesophagus/Lung/Heart)
   * Critical fail banner (with “what failed”)
   * Benchmark/tier indicator (Elite/Expert/Proficient/Developing as per your UI)
3. Keyboard ergonomics:

   * Tab order by clinical priority (PTV → cord/PRV → oesophagus → lung → heart)
   * Enter triggers recalculation (if not already live)
   * Ctrl+S saves attempt (if valid)
4. Validation:

   * Range checks per field
   * Cross-field warnings (V4.5 ≥ V16; V19.5 ≥ V39; percent fraction detection)
   * If “Not achievable” toggled: require a short reason (stored locally)
5. Save attempt CTA:

   * Primary “Save attempt” button
   * Show “unsaved changes” state
   * On save, show timestamp and new personal best if applicable

Acceptance criteria

* Entire metric set is viewable with minimal scrolling on a typical desktop monitor.
* Users can iterate rapidly and see where they’re losing points.
* Critical failures are unmissable and explain what to fix.
* All validations show clear messages (units included).
* No patient identifiers can be saved in notes.

4. Export/Import:

   * CSV export of attempts
   * JSON export/import for portability (optional)

Acceptance criteria

* Export files contain only metrics + scores + timestamps (no IDs).
* Reset clears all local data safely.
